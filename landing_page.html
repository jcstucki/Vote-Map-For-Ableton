BYPASS ROOM CODE

What is this?

This is the landing page for VoteMap, a live performance tool built for MaxMSP and Max 4 Live (Ableton).

From here you can enter a room number given to you by your host/performer, and be led to a page where you are able to democratically vote for parameters that the host has assigned in their Max patch or Ableton performance.

With access to the Max patch, or M4L Ableton device, you are able to map (link) values voted on by clients to parameters within your Max patch or Ableton set.

For example, clients can vote between three given options (ex. red/green/blue) which are then assigned within Max/Ableton to the output level of three oscillators with different timbres.

This allows for direct, near-instantaneous, interaction between audience, and performer.




How does this work?

VoteMap's server architecture is unix based, running on a DigitalOcean droplet. It's webserver is nginx, with a node.js backend. Node.js was chosen due to its integration with Max/MSP via the node.script max object, and it's event driven architecture. Communication with Max, the webserver, and clients is processed by socket.io, providing fast, efficient, event based communication between the three(+) nodes.

For each "room," each client is allowed one(1) vote per "section", with multiple "sections" being configured by the Max host application (soonTM). Clients may vote at any time, and change their vote as often or as little as they like.

The server keeps track of the unique client id, given by socket.io. Each updated/new vote, the server counts current votes and calculates a normalized percentage, which is then sent to the Max/M4L device that is the "host" of the room. In reality, the Max device is just another client, with special functionality, and it's presence does not count towards the vote percentage.

From there, it is simply up to the max user to assign the values output by the script(within Max), or assign the values to parameters via the GUI (M4L device) via the simple Map button on the GUI.




When was this made?

The initial concept was thought up during the 2019-2020 COVID pandemic, with a working implementation being created in Jan/Feb 2021. It is loosely based on another Max device by the author, "API2MIDI" which allowed mapping of comments on the website Reddit, to MIDI notes and data.




Where can this be used?

Clients can connect from anywhere where there is a live web-connection, as the only data being downloaded from the server is basic HTML/CSS/PHP and the socket.io framework for Javascript. It is so lightweight, that clients can connect via their phones, making this an incredibly powerful tool for live performance. (Hey! Finally, a good use of phones during concerts!)




Why was this made?

Because I hadn't heard of anything like it, and I haven't seen any musicians or artists utilizing phones in this manner.

"It's the new thing mannnnnn..."

There has always been a history of audience-participation in musical performance. From call & response, to "section" based assignment of vocals, crowdsurfing musicians, to "Play Freebird!" chants. While these examples use the musicians as an intermediary, they are more static than what this tool offers. VoteMap offers a "closer" to dynamic music experience than the previous examples.

Typically, these participatory music experiences do not cater to the individual audience member, only to the broader crowd, and do not allow those who appreciate music but do not have any musical tallent to create music.

With this tool, the non-musically inclined, can directly interact with the song/performance (this can even be used for lights), and say that they had a direct involvement with the performance (more direct than previous examples, I argue).

Each performance will be different, each song unique to the time and place. No singular song be repeated perfectly every time.

You'll never know what you'll get (mostly ;) )
